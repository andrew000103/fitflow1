# [Design] 페르소나 캐릭터 시스템 (Persona Character System)

## 1. 개요 (Executive Summary)
본 설계서는 사용자의 활동 데이터와 온보딩 정보를 함께 활용해 페르소나를 도출하고, 이를 홈 화면에 시각화하는 구체적인 로직과 UI 구조를 정의합니다.

### 구현 현황 / 운영 주의
- 현재 페르소나 시스템은 온보딩 + 최근 행동 데이터를 조합한 **휴리스틱 추정**이며, 사용자의 고정된 정체성을 판정하지 않는다.
- 기록이 부족하거나 신호가 충돌하면 더 보수적이고 중립적인 후보를 우선한다. 노출 문구도 **best-effort 안내**로 유지한다.
- 상태값과 캐릭터 매칭은 데이터 재계산 시점에 따라 달라질 수 있으므로, 운영/QA에서는 단일 정답을 강하게 보장하지 않는다.

| 분류 | 내용 |
|---|---|
| **핵심 로직** | 온보딩 기반 초기 추정 + 최근 14일 활동 기반 보정 → 12종 페르소나 매칭 |
| **콜드스타트 대응** | 기록 부족 시 `Starter` 단계로 시작하고, 데이터가 쌓일수록 행동 기반 분류 비중 확대 |
| **상태 변화** | 당일 활동(운동 완료, 칼로리 달성도)에 따른 캐릭터 상태 피드백 |
| **UI 통합** | 홈 화면 `CalorieRing` 중앙에 캐릭터 배치, 상단에 별명 및 상태 메시지 표시 |

## 2. 페르소나 도출 로직 (Persona Engine)

### A. 설계 원칙 (Cold Start First)
- 신규 사용자는 최근 14일 운동 기록, 최근 7일 식단 기록이 충분하지 않을 수 있으므로, 초기 1~2주는 "정확한 분류"보다 "부담 없는 적응"을 우선한다.
- 초반에는 사용자의 실제 습관을 단정하지 않는다. `Sporadic`, `Cheater` 같은 부정적 해석이 가능한 라벨은 충분한 데이터가 쌓이기 전까지 노출하지 않는다.
- 페르소나는 고정 정체성이 아니라 "현재 추정 상태"로 간주하며, 기록이 쌓이면 자연스럽게 업데이트된다.

### B. 사용자 단계 정의 (Persona Lifecycle)
1. **`starter` 단계**
   - 조건: 가입 후 7일 미만 또는 운동 기록 3회 미만 또는 식단 기록 3일 미만
   - 동작: 온보딩 기반 추정 페르소나만 사용
   - 노출 문구 예시: "지금은 당신의 루틴을 배우는 중이에요"
2. **`learning` 단계**
   - 조건: 가입 후 7일 이상이며 운동 기록 3~9회 또는 식단 기록 3~6일
   - 동작: 온보딩 + 실제 기록 혼합 점수 사용
   - 노출 문구 예시: "당신의 패턴이 조금씩 보이기 시작했어요"
3. **`established` 단계**
   - 조건: 최근 14일 운동 기록 10회 이상 또는 최근 7일 식단 기록 7일 확보
   - 동작: 실제 기록 기반 점수 비중을 최대로 적용
   - 노출 문구 예시: "이제 당신의 루틴이 꽤 선명해졌어요"

### C. 차원별 점수 산출 (Dimension Scoring)
1. **운동 빈도 (Frequency Score, `F`):** 최근 14일 중 운동 세션 수
   - `F >= 10`: 하드코어 (Hardcore)
   - `4 <= F < 10`: 꾸준함 (Steady)
   - `F < 4`: 간헐적 (Sporadic)
2. **식단 성격 (Diet Score, `D`):** 최근 7일 평균 단백질 섭취 비율 (목표 대비) 및 칼로리 준수율
   - `D_avg >= 0.9`: 단백질 중시 (Protein)
   - `0.7 <= D_avg < 0.9`: 균형 잡힌 (Balanced)
   - `D_avg < 0.7`: 치팅 빈번 (Cheater)
3. **목표 (Goal, `G`):** `user_goals.goal_type` 사용
   - `gain`: 벌크업/증량
   - `maintain`: 유지/건강
   - `loss`: 다이어트/감량

### D. 초기 추정 점수 (Bootstrap Scoring)
기록이 부족할 때는 온보딩 데이터를 사용해 임시 점수를 계산한다.

1. **초기 운동 성향 (`F_bootstrap`)**
   - `workoutDaysPerWeek >= 5` 또는 `experience = advanced`: Hardcore 후보
   - `workoutDaysPerWeek 3~4`: Steady 후보
   - `workoutDaysPerWeek <= 2` 또는 `experience = beginner`: Starter/Steady 후보
2. **초기 식단 성향 (`D_bootstrap`)**
   - `goal = muscle_gain` 또는 `strength_gain`이고 식이 제한이 적음: Protein/Balanced 후보
   - `goal = maintenance` 또는 `health`: Balanced 후보
   - `overeatingHabit = often`이면 초기 상태 메시지에만 반영하고, `Cheater` 라벨에는 직접 연결하지 않음
3. **목표 (`G`)**
   - `OnboardingData.goal`을 우선 사용하고, 이후 실제 `user_goals.goal_type`과 동기화

### E. 점수 혼합 규칙 (Blended Scoring)
충분한 로그가 쌓이기 전까지는 초기 추정 점수와 실제 기록 점수를 혼합한다.

| 단계 | 온보딩 비중 | 행동 데이터 비중 |
|---|---:|---:|
| `starter` | 0.8 | 0.2 |
| `learning` | 0.5 | 0.5 |
| `established` | 0.2 | 0.8 |

- 운동 로그가 0건이면 `F`는 `F_bootstrap`을 그대로 사용한다.
- 식단 로그가 0일이면 `D`는 `D_bootstrap`을 그대로 사용한다.
- 특정 축의 데이터만 부족하면 해당 축만 bootstrap 값을 사용하고, 다른 축은 실제 기록을 반영한다.

### F. 페르소나 매트릭스 (Mapping Matrix)

| 목표 (G) | 성실도 (F) | 식단 (D) | 캐릭터 (예시) | 별명 (Nickname) |
|---|---|---|---|---|
| **Gain** | Hardcore | Protein | **고릴라** | 정진하는 고릴라 |
| **Gain** | Steady | Balanced | **곰** | 묵직한 불곰 |
| **Loss** | Hardcore | Protein | **치타** | 날카로운 치타 |
| **Loss** | Steady | Balanced | **사슴** | 우아한 사슴 |
| **Maintain** | Steady | Balanced | **강아지** | 활기찬 리트리버 |
| **Maintain** | Sporadic | Cheater | **나무늘보** | 여유로운 나무늘보 |
| ... | ... | ... | (추가 6종) | ... |

### G. 초기 사용자 보호 규칙 (Beginner Guardrails)
- `starter` 단계에서는 `Sporadic`, `Cheater`, `나무늘보` 같은 저의욕 인상을 줄 수 있는 페르소나를 배정하지 않는다.
- `starter` 단계 기본 후보군은 `강아지`, `곰`, `사슴`처럼 중립적이거나 격려형 인상의 캐릭터로 제한한다.
- 첫 3회 운동 완료 전에는 "게으름", "치팅", "불규칙"을 암시하는 상태 메시지를 금지한다.
- 첫 7일 동안 식단 로그가 적더라도 `HUNGRY`, `FULL` 판정은 "정보 부족" 배지를 함께 노출하거나 민감도를 완화한다.

## 3. 당일 상태 피드백 로직 (Daily State)

캐릭터의 일러스트는 `PersonaID`와 `State`의 조합으로 결정됩니다.

- **`ACTIVE`**: 오늘 운동 완료 + 칼로리 섭취 80~110% (최고의 상태)
- **`RESTING`**: 운동 쉬는 날 + 칼로리 섭취 90~110% (안정적인 상태)
- **`HUNGRY`**: 칼로리 섭취 < 70% (배고픈 상태)
- **`FULL`**: 칼로리 섭취 > 120% (배부른 상태)
- **`TIRED`**: 오늘 운동 완료했으나 단백질 섭취 부족 (회복 필요 상태)

### A. 초기 사용자 예외 처리
- 가입 후 첫 3일은 당일 상태를 너무 공격적으로 해석하지 않는다.
- 식단 로그가 1끼 이하인 날에는 `HUNGRY` 대신 `LOG_MORE` 내부 상태를 사용하고, UI에는 "오늘 식사를 조금 더 기록해볼까요?"처럼 안내형 메시지를 보여준다.
- 오늘 운동 로그가 없더라도 계획된 휴식일 여부를 알 수 없으면 `RESTING`을 기본값으로 사용한다.
- `TIRED`는 단백질 목표와 실제 섭취 로그가 모두 충분히 존재하는 날에만 활성화한다.

## 4. UI/UX 및 컴포넌트 구조

### A. 컴포넌트 배치 (`HomeScreen.tsx`)
1. **`PersonaHeader`**: "정진하는 고릴라 님, 오늘도 득근하세요!" (별명 + 말풍선 메시지)
   - `starter` 단계에서는 "정체성 확정"보다 "성장 중" 뉘앙스를 사용
   - 예시: "리듬을 찾는 리트리버", "당신의 루틴을 배우는 중이에요"
2. **`PersonaCharacter`**: `CalorieRing`의 `Svg` 중앙 영역 또는 상단 오버레이 영역에 배치
   - 현재 `CalorieRing` 중앙에 칼로리 숫자가 고정 배치되어 있으므로, 1차 구현에서는 중앙 전면 교체보다 작은 배지/오버레이 형태를 우선 검토
   - `starter` 단계에서는 전체 캐릭터 대신 작은 알/새싹/실루엣 형태를 사용하면 초기 추정이라는 메시지가 자연스럽다
   - 클릭 시 "오늘의 조언" 팝업

### B. 상태 관리 (`persona-store.ts`)
```typescript
interface PersonaState {
  currentPersonaId: string;
  personaStage: 'starter' | 'learning' | 'established';
  confidence: number; // 0~1
  dailyState: 'ACTIVE' | 'RESTING' | 'HUNGRY' | 'FULL' | 'TIRED';
  lastUpdated: string;
  calculatePersona: (userId: string) => Promise<void>;
}
```

### C. 메시지 톤 가이드
- 기록이 부족할수록 평가형 문장보다 코칭형 문장을 사용한다.
- 나쁜 습관을 단정하지 않고, "아직 패턴을 파악 중"이라는 표현을 우선한다.
- 예시
  - `starter`: "첫 주는 가볍게 기록만 해도 충분해요"
  - `learning`: "최근엔 저녁 운동 루틴이 조금씩 보이네요"
  - `established`: "단백질과 운동 루틴이 꽤 안정적으로 자리잡았어요"

## 5. 데이터베이스 및 에셋 (Assets)
- **에셋 경로**: `assets/images/persona/{personaId}_{state}.png`
- **로직 위치**: `src/lib/persona-engine.ts` (점수 계산 순수 함수)
- **추가 메타데이터**: 퍼소나 계산 시 `dataCompleteness`, `personaStage`, `confidence`, `sourceBreakdown`을 함께 저장하면 디버깅과 A/B 테스트에 유리하다.

## 6. 검증 (Validation)
- 최근 14일간의 운동/식단 Mock 데이터를 통해 12종 페르소나가 기대 범위 안에서 도출되는지 확인한다.
- 캐릭터 배치가 `CalorieRing` 내부 텍스트와 겹치지 않도록 레이아웃을 점검한다.
- 가입 첫날, 운동만 있는 사용자, 식단만 있는 사용자, 기록이 전혀 없는 사용자를 각각 수동으로 확인한다.
- 초기 7일 동안 부정적 라벨이 노출되지 않는지 확인한다.
- 행동 데이터 증가에 따라 `starter -> learning -> established` 전이가 자연스럽게 일어나는지 확인한다.
- 운영 관점에서는 페르소나가 `정답`으로 고정되지 않고, 재계산 시 결과가 바뀔 수 있음을 전제로 검증한다.
- 수동 체크리스트
  - 신규 가입 직후 홈 진입 시 중립적/격려형 문구만 노출되는지 확인한다.
  - 운동 로그만 존재하는 계정에서 식단 부족 상태가 과하게 단정되지 않는지 확인한다.
  - 식단 로그만 존재하는 계정에서 운동 성향이 과도하게 확정되지 않는지 확인한다.
  - 기록 공백, 프로필 변경, 재로그인 후에도 캐릭터와 상태가 깨지지 않는지 확인한다.
  - 홈 화면에서 상태 문구가 진단처럼 보이지 않고 안내형 톤을 유지하는지 확인한다.
