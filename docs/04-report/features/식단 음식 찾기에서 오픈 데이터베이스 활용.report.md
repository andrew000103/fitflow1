#### 1. diet-search.ts (+30줄)

```typescript
// 추가 import
import { FoodItem } from '../types/food';

// saveMealItemFromFoodItem 함수
const UUID_REGEX = /^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$/i;

export async function saveMealItemFromFoodItem(params: {
  userId: string;
  foodItem: FoodItem;
  mealType: string;
  date: string;
  amountG: number;
}): Promise<void> {
  const { userId, foodItem, mealType, date, amountG } = params;
  const ratio = amountG / 100;
  const n = foodItem.nutrients;

  const { error } = await supabase.from('meal_items').insert({
    user_id: userId,
    food_id: UUID_REGEX.test(foodItem.id) ? foodItem.id : null,
    meal_type: mealType,
    date,
    amount_g: amountG,
    calories: Math.round(n.calories * ratio),
    protein_g: Math.round(n.protein_g * ratio * 10) / 10,
    carbs_g: Math.round(n.carbs_g * ratio * 10) / 10,
    fat_g: Math.round(n.fat_g * ratio * 10) / 10,
    sodium_mg: n.sodium_mg != null ? Math.round(n.sodium_mg * ratio * 10) / 10 : null,
    sugar_g: n.sugar_g != null ? Math.round(n.sugar_g * ratio * 10) / 10 : null,
  });

  if (error) throw error;
}
```

**특징**:
- UUID_REGEX: UUID 형식 검사 (Supabase custom food ID)
- Non-UUID (OFFs, MFDS, USDA): `food_id = null`
- 영양소 계산: per 100g 기준 (기존과 동일)

  const normalized = q.trim();

  }
  if (pg === 1) {
    setLoading(true);
    setHasSearched(false);

    setLoadingMore(true);

  try {
    const items = await searchFoods(normalized, pg, user?.id ?? undefined);
    if (pg === 1) setResults(items);
    else setResults(prev => [...prev, ...items]);
    setPage(pg);
  } catch {
    if (pg === 1) setResults([]);
  } finally {
    if (pg === 1) {
      setLoading(false);
      setHasSearched(true);
    } else {
      setLoadingMore(false);
    }
  }
}, [user?.id]);

## 학습 사항 (Lessons Learned)

### 1. 타입 전환의 중요성

**상황**: FoodRow → FoodItem 타입 전환

**배운 점**:
- 기존 코드에서 FoodRow 타입 사용
- 신규 food-search.ts가 FoodItem 타입 제공
- 타입 불일치로 인한 컴파일 오류 예상

**해결 방법**:
- 이미 구현된 `toFoodItem(FoodRow): FoodItem` 변환 함수 활용
- 화면 컴포넌트의 상태를 FoodItem으로 일원화
- FoodRow는 직접 입력 후 변환할 때만 사용

**재사용 가능한 패턴**:
> 기존 타입과 신규 타입이 공존할 때, 변환 함수로 경계를 명확히 하면 단계적 리팩토링 가능. 전체 코드를 한 번에 변환하지 않고, 입출력 경계에서만 변환하면 리스크 감소.

### 2. 페이지네이션 상태 관리

**상황**: "더 보기" 버튼으로 추가 결과 로드

**배운 점**:
- page state + loadingMore flag 분리의 필요성
- 새 검색어 입력 시 page 초기화 필수
- 로딩 중 중복 클릭 방지

**해결 방법**:
- `page` state: 현재 페이지 번호 추적
- `loadingMore` flag: 로딩 상태 (버튼 비활성화)
- `doSearch(q, pg)`: pg=1일 때 교체, pg>1일 때 append
- useEffect debounce: 검색어 변경 시 page=1 초기화

**재사용 가능한 패턴**:
> 무한 스크롤/페이지네이션은 page state와 로딩 flag를 분리하면 관리가 명확. 검색어 변경 시 page 초기화는 useEffect에서 처리하는 것이 더 안전.

### 3. 출처별 뱃지 설계

**상황**: 4종 데이터 소스(custom/mfds/openfoodfacts/usda) 구분 필요

**배운 점**:
- 각 소스별로 서로 다른 색상/라벨 필요
- 다크 모드에서도 readable한 색상 조합 필요

**해결 방법**:
- SOURCE_BADGE_CONFIG: 소스별 { label, bgColor, textColor } 맵 생성
- 연한 배경 + 진한 텍스트 조합 (다크 모드 호환)
- `useAppTheme` 의존 없이 하드코딩으로 명시성 확보

**재사용 가능한 패턴**:
> 다중 소스/카테고리를 시각화할 때, 설정 객체(Record)로 일관성 있게 관리. useAppTheme 의존이 필수 아닌 경우 하드코딩으로 명시성 향상.

### 4. 신규 DB 모듈로의 마이그레이션 전략

**상황**: diet-search.ts (구형 2종) → food-search.ts (신형 4종) 전환

**배운 점**:
- 신규 모듈이 이미 구현되어 있었으나 화면에서 미연결
- 직접 import만 변경하면 자동으로 4종 검색 활성화
- API 키 없어도 OFFs는 항상 동작

**해결 방법**:
- import { searchFoods } from '../../lib/food-search' 로 변경
- 타입 전환 (FoodRow → FoodItem)
- 나머지 로직은 그대로 재사용

**재사용 가능한 패턴**:
