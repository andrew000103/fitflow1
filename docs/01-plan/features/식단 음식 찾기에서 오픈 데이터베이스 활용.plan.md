**Date**: 2026-03-23

## Executive Summary

| 관점 | 내용 |
|------|------|
| **Problem** | `food-search-screen.tsx`가 구형 `diet-search.ts`(Supabase + OFFs 2종)만 사용하고, 이미 구현된 4종 통합 `food-search.ts`(식약처 MFDS + OFFs + USDA + 커스텀)에 연결되지 않아 검색 결과가 빈약하다. |
| **Solution** | 화면을 신형 `food-search.ts`로 전환하고, `FoodItem` 타입 기반으로 리팩토링하며 출처 뱃지와 "더 보기" 페이지네이션을 추가한다. |
| **Function UX Effect** | 음식 검색 시 OFFs(무료) 데이터가 즉시 표시되고, 출처 뱃지로 어느 DB에서 왔는지 확인 가능하며, "더 보기"로 추가 결과를 로드할 수 있다. |
| **Core Value** | API 키 없이도 OFFs 오픈 DB를 통해 광범위한 식품 데이터를 검색할 수 있고, 향후 MFDS/USDA 키 설정 시 자동으로 한국/미국 DB가 활성화되는 확장 가능한 구조. |

## 1. 배경 및 문제 정의

### 1.1 현재 상태

**두 개의 food search 모듈이 공존:**

| 파일 | 사용 화면 | 제공자 | 타입 |
|------|----------|--------|------|
| `diet-search.ts` | `food-search-screen.tsx` (현재) | Supabase DB + OFFs | `FoodRow` |
| `food-search.ts` | **미연결** (사용 화면 없음) | 커스텀 + MFDS + OFFs + USDA | `FoodItem` |

- `food-search.ts`는 4종 통합, 24시간 로컬 캐시, deduplication, provider timeout(5s), 페이지네이션을 모두 구현
- `food-search-screen.tsx`는 여전히 `diet-search.ts`에서 `FoodRow` 타입으로 데이터를 받아 처리
- 타입 불일치: `diet-search.ts`는 `FoodRow`, `food-search.ts`는 `FoodItem`(diet-store 표준 타입)

### 1.2 API 키 현황

- MFDS(식약처), USDA: `.env`에 API 키 미설정 → 해당 provider는 빈 배열 반환(silently skip)
- OFFs: 무료, API 키 불필요 → 항상 동작
- 커스텀 음식: AsyncStorage 기반, 항상 동작

### 1.3 목표

1. `food-search-screen.tsx`를 `food-search.ts` 기반으로 전환
2. `FoodItem` 타입 기반으로 화면 로직 통일
3. 출처 뱃지(식약처/OFFs/USDA/내 음식) UI 추가
4. "더 보기" 페이지네이션 추가

## 2. 요구사항

### 2.1 기능 요구사항

#### FR-01: food-search-screen → food-search.ts 전환

- `import { searchFoods } from '../../lib/diet-search'` → `import { searchFoods } from '../../lib/food-search'`
- `FoodRow` 타입 의존성 제거, `FoodItem` 타입으로 통일
- `AmountModal`: `FoodRow` → `FoodItem` 기반으로 영양소 계산 방식 변경
  - `food.calories_per_100g` → `food.nutrients.calories`
  - `food.protein_per_100g` → `food.nutrients.protein_g`

#### FR-02: FoodItem 기반 meal_items 저장

- 기존 `saveMealItem(params: SaveMealItemParams)`은 `FoodRow` 기반
- `diet-search.ts`에 `saveMealItemFromFoodItem` 함수 추가:
  ```typescript
  export async function saveMealItemFromFoodItem(params: {
    userId: string;
    foodItem: FoodItem;
    mealType: string;
    date: string;
    amountG: number;
  }): Promise<void>
  ```
- `food-search-screen.tsx`는 이 새 함수를 사용

#### FR-03: 출처 뱃지 표시

- 검색 결과 각 행에 출처 뱃지 추가
- `FoodItem.source` 값에 따른 뱃지:

  | source | 뱃지 텍스트 | 색상 |
  |--------|------------|------|
  | `'custom'` | 내 음식 | accent |
  | `'mfds'` | 식약처 | green |
  | `'openfoodfacts'` | OFFs | blue |
  | `'usda'` | USDA | orange |

- 뱃지: 작은 pill 형태, 항목명 오른쪽에 배치

#### FR-04: "더 보기" 페이지네이션

- `searchFoods(query, page, userId)` — `page` 파라미터 활용
- 첫 검색: `page=1`, 결과 하단에 "더 보기" 버튼 표시
- "더 보기" 탭 시: `page+1`로 호출, 기존 결과에 append
- 로딩 중 버튼 비활성화 + ActivityIndicator 표시
- 새 검색어 입력 시 page 초기화 (page=1)

### 2.2 비기능 요구사항

- `food-search.ts`의 기존 캐시(24시간), deduplication, timeout 로직은 그대로 활용
- 기존 diet-store의 `addEntry` 흐름 유지 (FoodItem 타입 이미 호환)
- MFDS/USDA API 키 없어도 OFFs 결과 정상 표시
