# AI Plan Loading UX — Plan Document

> **작성일**: 2026-03-25
> **Phase**: Plan
> **Status**: Draft

---

## Executive Summary

| 관점 | 내용 |
|------|------|
| **Problem** | AI 플랜 생성에 10~30초 소요. 현재 화면은 이모지 + 텍스트 2줄뿐이라 사용자가 지루해서 앱을 꺼버리고 플랜을 받지 못함 |
| **Solution** | 대기 시간을 "앱 학습 시간"으로 전환. 단계별 진행 메시지 + 앱 기능 소개 슬라이드 + 운동/식단 팁 카드로 로딩 화면 풍성하게 구성 |
| **Function UX Effect** | 로딩 중 이탈률 감소. 사용자가 앱 기능을 자연스럽게 학습하고, 팁으로 동기부여 유지 |
| **Core Value** | 대기 시간을 비용이 아닌 가치로 전환 — 처음 사용자가 앱을 이해하는 핵심 접점 |

## Context Anchor

| 항목 | 내용 |
|------|------|
| **WHY** | 플랜 생성 대기 이탈 → 피처 효용 0. 로딩을 "뭔가 되고 있다"는 확신과 "앱이 재미있다"는 인상으로 전환 |
| **WHO** | AI 플랜 온보딩 완료 후 처음 생성하는 신규 사용자 (특히 앱에 익숙하지 않은 입문자) |
| **RISK** | 카드 UI 구현 복잡도 상승 / 타이머와 실제 API 완료 타이밍 불일치 시 UX 어색함 |
| **SUCCESS** | 로딩 화면에서 사용자가 스크롤/스와이프 인터랙션 1회 이상 발생 (참여 확인). 앱 종료 없이 플랜 결과 화면까지 도달 |
| **SCOPE** | P0: 로딩 화면 UX 개선 (C+A+E). P1: 백그라운드 생성 + 푸시 알림 (B, 별도 피처) |

## 1. 배경 및 문제 정의

### 1.1 현재 상황

```
[온보딩 완료] → handleFinish() → API 호출 시작
                                      ↓
                        로딩 화면: 🤖 "AI가 플랜을 만들고 있어요..."
                                      ↓ (10~30초)
                        결과 화면 자동 이동
```

**문제점**:
- 진행 상황을 전혀 알 수 없음 → "멈춘 건지?" 불안감
- 할 게 없음 → 이탈 유도
- 첫 사용자 온보딩 기회 낭비

### 1.2 목표 상황

```
[온보딩 완료] → API 호출 시작
                     ↓
        ┌─────────────────────────────┐
        │  🔍 신체 정보 분석 중... ✓  │  ← 단계별 진행 (C)
        │  🍽️ 식단 구성 중...         │
        │  💪 운동 계획 수립 중        │
        ├─────────────────────────────┤
        │  ← 슬라이드 카드 →          │  ← 튜토리얼(A) + 팁(E) 혼합
        │  앱 기능 소개 or 운동 팁    │
        │  3초 자동 전환, 스와이프 가능│
        └─────────────────────────────┘
                     ↓ (API 완료)
              결과 화면 자동 이동
```

## 2. 요구사항

### 2.1 P0 — 빠른 구현 (이번 피처)

#### C. 단계별 진행 메시지 (타이머 기반 시뮬레이션)

| 단계 | 아이콘 | 메시지 | 시작 시간 |
|------|--------|--------|:---------:|
| 1 | 🔍 | 신체 정보 분석 중... | 0s |
| 2 | 🍽️ | 식단 구성 중... | 5s |
| 3 | 💪 | 운동 계획 수립 중... | 10s |
| 4 | ✨ | 플랜 최적화 중... | 16s |
| 5 | 🎯 | 마무리 중... | 22s+ (루프) |

- 완료된 단계: 체크마크(✓) + 흐릿하게
- 현재 단계: 강조 + 점점이 애니메이션 (`...`)
- API가 빨리 끝나면 현재 단계에서 즉시 이동

#### A. 앱 기능 소개 슬라이드 (4장)

| 슬라이드 | 제목 | 설명 | 아이콘 |
|---------|------|------|--------|
| 1 | 운동 기록 | 세트·반복·중량을 한 번에. 휴식 타이머 자동 시작 | 💪 |
| 2 | 식단 검색 | 국내외 식품 DB + 직접 입력. 매크로 자동 계산 | 🍽️ |
| 3 | 홈 대시보드 | 칼로리·운동·체중 변화를 한눈에 | 📊 |
| 4 | AI 주간 플랜 | 매주 월요일 새 플랜. 지난 주와 비교해서 진화 | 🤖 |

#### E. 운동/식단 팁 카드 (10개, 자동 전환 + 스와이프)

| # | 카테고리 | 팁 |
|---|---------|-----|
| 1 | 운동 | 근성장을 위한 최소 단백질: 체중(kg) × 1.6g/일 |
| 2 | 식단 | 물 하루 체중(kg) × 30ml. 공복감의 30%는 갈증 |
| 3 | 운동 | 점진적 과부하: 매주 중량 2.5~5% 증가가 목표 |
| 4 | 식단 | 단백질은 포만감 지속 시간이 탄수화물의 2배 |
| 5 | 운동 | 수면 7시간 이상이 테스토스테론 생성의 기본 조건 |
| 6 | 식단 | 식사 직전 채소 → 혈당 스파이크 20~30% 감소 |
| 7 | 운동 | 데드리프트·스쿼트 같은 다관절 운동이 단시간 효율 최고 |
| 8 | 식단 | 현미 vs 백미: 혈당지수 55 vs 72. 포만감 2배 차이 |
| 9 | 운동 | 근육통(DOMS)은 성장 신호가 아님. 회복의 신호 |
| 10 | 멘탈 | 3주 지속 시 습관 형성 시작. 처음 21일이 가장 중요 |

**카드 동작**:
- 3초마다 자동으로 다음 카드
- 하단 페이지 인디케이터 점(·) 표시
- 좌우 스와이프로 수동 이동 가능
- 튜토리얼 4장 + 팁 10장 = 총 14장 (무작위 섞기 or 순서대로)

## 4. 성공 기준

| 기준 | 측정 방법 |
|------|-----------|
| 로딩 화면에서 앱 종료 없이 결과 화면 도달 | 사용자 테스트 |
| 단계별 메시지가 API 완료 전에 자연스럽게 진행 | 30초 이상 API 응답 시에도 루프 |
| 카드 인터랙션 발생 (스와이프 or 자동 전환) | 구현 확인 |
| API 완료 즉시 자동으로 결과 화면 이동 | 기능 정상 동작 |

---

## 5. 제약 사항

- Gemini REST fetch는 단일 HTTP 요청 — 실시간 스트리밍 불가. 진행 메시지는 타이머 시뮬레이션만 가능
- React Native 메인 스레드에서 setInterval 사용 — 무거운 애니메이션과 병행 주의
- Expo managed workflow — 백그라운드 작업은 P1에서 별도 처리

## 6. 위험 요소

| 위험 | 대응 |
|------|------|
| API가 5초 이내 완료 | 단계 1만 진행 후 즉시 이동 — 자연스러움 |
| API가 60초 초과 | 타임아웃 에러 처리는 기존 코드에 이미 있음 |
| 카드 자동 전환이 사용자 스와이프와 충돌 | 스와이프 감지 시 자동 전환 일시 정지 |
