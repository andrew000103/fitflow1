## 1. Executive Summary

## 2. 타깃 페르소나

### Persona A — "정체기 남성 벌커"
- 헬스 경험 1년 이상, 루틴은 있지만 체중/근육량이 몇 달째 변화 없음
- nSuns 같은 구조화된 프로그램을 쓰고 있지만 식단이 들쭉날쭉
- 원하는 것: "내 상태에서 뭘 바꿔야 다시 느는지" 구체적 답변
- AI 기대치: 숫자로 설명해주는 전문가 느낌

### Persona B — "입문 여성 다이어터"
- 운동 경험 없음, 체중 감량 목표이나 굶는 방법밖에 모름
- 뭘 먹어야 하는지, 어떤 운동을 해야 하는지 전혀 감이 없음
- 원하는 것: "나 같은 사람도 할 수 있는 구체적인 계획"
- AI 기대치: 쉬운 말로 설명해주는 친근한 코치 느낌

## 3. 핵심 기능 목록

| 우선순위 | 기능 | 설명 |
|----------|------|------|
| P0 | 초기 AI 플랜 생성 | 온보딩 정보 기반으로 1주일치 식단+운동 계획 생성 |
| P0 | 정기 재생성 (주 1회) | 매주 자동으로 다음 주 계획 갱신 |
| P0 | 안전 가드레일 | 위험 목표 거부 + 경고 제공 |
| P1 | 계획 이유 설명 (접기/펼치기) | 왜 이 계획인지 숫자+비교 데이터+출처로 설명 |
| P1 | 데이터 동의 기반 히스토리 반영 | 동의 사용자: 최근 1주 데이터 AI 컨텍스트로 활용 |
| P2 | 정체기 감지 + 플랜 수정 제안 | 체중/운동 기록 패턴으로 정체기 자동 감지 |
| P2 | 프로필 내 AI 기능 온/오프 | 나중에 동의한 사람도 최근 1주 데이터 기반으로 활성화 |

## 4. 온보딩 데이터 수집

### 4-1. 필수 수집 (Phase 1 — 즉시)

| 항목 | 입력 방식 | 비고 |
|------|-----------|------|
| 나이 | 숫자 입력 | |
| 성별 | 선택 | 남/여/밝히지 않음 |
| 키 / 체중 | 숫자 입력 | 기존 프로필 연동 가능 |
| 주요 목표 | 선택 | 체중 감량 / 근육 증가 / 체형 유지 / 건강 개선 |
| 운동 경험 | 선택 | 입문(0~6개월) / 초급(6개월~2년) / 중급(2년+) |
| 주당 운동 가능 일수 | 선택 (1~7일) | |
| 식이 제한 | 다중 선택 | 채식 / 유제품 제외 / 글루텐 제외 / 없음 |

### 4-2. 우회 질문으로 수집 (Phase 2 — 프로그레시브)

> 객관적 수치로 표현하기 어려운 상태를 간접 질문으로 파악

| 파악하려는 상태 | 우회 질문 예시 |
|----------------|----------------|
| 회복 능력 / 피로도 | "최근 운동 다음 날 일상생활이 얼마나 불편한가요?" (전혀 / 약간 / 많이) |
| 식욕 조절 능력 | "배고프지 않아도 습관적으로 먹는 편인가요?" |
| 수면 품질 | "일어났을 때 피로가 회복된 느낌인가요?" (대부분 / 가끔 / 거의 안 됨) |
| 스트레스 레벨 | "최근 한 달간 스트레스가 식욕이나 운동 의지에 영향을 줬나요?" |
| 과거 정체기 경험 | "이전에 비슷한 목표를 시도했다가 포기한 적 있나요? 이유는?" |

### 4-3. 기존 앱 데이터 활용 (동의자 한정)

- 최근 7일간 `workout_sessions`, `meal_items`, `body_weights` 요약
- AI 프롬프트 컨텍스트에 포함: 평균 칼로리 섭취, 운동 완료율, 체중 변화 추이
- 동의 거부 시: 수동 입력 데이터만 사용

## 5. UX 흐름

### 5-1. 온보딩 플로우 (앱 첫 실행)

```
앱 첫 실행
    ↓
[기본 회원가입/로그인]
    ↓
"AI 플랜 기능을 활성화할까요?"
├── 동의 → 데이터 활용 범위 설명 → 온보딩 질문 → AI 플랜 생성 → 홈
└── 거부 → 일반 홈 진입 (프로필 > AI 기능에서 나중에 켜기 가능)
```

### 5-2. AI 플랜 생성 화면 흐름

```
온보딩 질문 (Phase 1: 필수 7항목)
    ↓
우회 질문 (Phase 2: 선택적 5항목, 스킵 가능)
    ↓
안전 검증 (위험 목표 체크)
├── 통과 → AI 플랜 생성 (로딩 ~5초)
└── 거부 → 경고 팝업 + "이 목표로는 AI 플랜을 생성할 수 없습니다" + 수정 유도
    ↓
플랜 결과 화면
├── 이번 주 운동 계획 (요일별)
├── 이번 주 식단 계획 (끼니별 예시)
├── 목표 칼로리/매크로 수치
└── [이유 보기] 접기/펼치기 섹션
```

### 5-3. 주간 재생성 흐름

```
매주 월요일 오전 알림
    ↓
"지난주 기록을 분석해 새로운 플랜을 준비했어요"
    ↓
홈 카드 탭 → 새 플랜 확인
    ↓
[적용하기] / [지난 주 플랜 유지]
```

## 6. 이유 설명 (Explainability) 설계

### 원칙
- 전문용어 최소화, 숫자와 비교 데이터 중심
- 출처 제공: 목표에 따라 다른 레퍼런스 제시

### 출처 전략

| 목표 | 출처 유형 예시 |
|------|----------------|
| 근육 증가 (벌크업) | "프로 보디빌더 Cbum의 오프시즌 매크로 비율 참고", "근비대 관련 메타 분석 (Schoenfeld, 2010)" |
| 체중 감량 | "건강하게 감량한 연예인 ○○의 식단 원칙 참고", "NASM 칼로리 적자 가이드라인" |
| 건강 유지 | "WHO 신체 활동 권장량 기반" |

### 설명 카드 예시

```
[왜 이 플랜인가요?] ▼

• 당신의 TDEE는 약 2,450kcal입니다
• 벌크업 목표이므로 +250~300kcal 서플러스 설정 → 목표 2,700kcal
• 단백질은 체중 1kg당 1.8~2g 기준 → 일 130g 목표
  (근비대 메타분석 기준, Schoenfeld 2010)
• 지난 주 평균 섭취 칼로리: 2,100kcal → 현재 약 350kcal 부족 상태
```

## 7. 안전 가드레일

### 거부 조건 (AI 계획 생성 불가)

| 조건 | 처리 방식 |
|------|-----------|
| 일일 칼로리 목표 < 1,200kcal (여성) / < 1,500kcal (남성) | 거부 + 위험성 설명 |
| 주당 체중 감량 목표 > 1kg | 경고 + 수정 유도 |
| BMI < 17.5 (저체중 상태에서 감량 목표) | 거부 + 전문가 상담 권고 |
| 임신 중 / 특정 질환 언급 (당뇨, 신장 질환 등) | 거부 + "의사·영양사 상담 필요" 안내 |

### 경고 메시지 원칙
- "이 목표는 건강에 위험할 수 있습니다. AI 플랜을 생성하려면 목표를 조정해주세요."
- 수정 버튼 제공 (목표 재입력 유도)
- 정보성 팝업: 해당 위험 요소에 대한 간략한 설명 (예: 극단적 칼로리 적자의 근손실 위험)

## 8. AI 기술 스택 및 구현 방향

### 8-1. LLM 선택지

| 옵션 | 장점 | 단점 | 추천 여부 |
|------|------|------|-----------|
| Claude API (claude-sonnet-4-6) | 긴 컨텍스트, 안전 필터링 강함, 한국어 품질 좋음 | 상대적 비용 | ⭐ 추천 |
| OpenAI GPT-4o | 빠른 응답, 구조화 출력 우수 | 프라이버시 우려 있음 | 대안 |
| GPT-4o-mini | 비용 저렴 | 품질 타협 | 비용 최적화 시 |

### 8-2. 프롬프트 구조

```
[시스템 프롬프트]
- 역할: 공인 영양사 + 개인 트레이너
- 제약: 위험 목표 거부 조건 명시
- 출력 형식: JSON (운동 계획 배열 + 식단 계획 배열 + 설명 텍스트)
- 언어: 한국어, 전문용어 최소화

[사용자 컨텍스트]
- 온보딩 데이터 (나이, 성별, 체중, 목표 등)
- 우회 질문 응답 요약
- [동의자] 최근 7일 운동/식단/체중 데이터 요약

[요청]
- 1주일치 운동 계획 (요일별 종목, 세트/반복 범위)
- 1주일치 식단 계획 (하루 칼로리, 매크로, 끼니별 예시 메뉴)
- 목표 칼로리/매크로
- 이유 설명 (숫자 + 출처 포함)
```

### 8-3. 응답 스키마 (JSON)

```typescript
interface AIPlan {
  generatedAt: string;
  targetCalories: number;
  targetMacros: { protein: number; carbs: number; fat: number };
  weeklyWorkout: WorkoutDay[];
  weeklyDiet: DietDay[];
  explanation: {
    summary: string;        // 한 줄 요약
    detail: string;         // 접기/펼치기용 상세 설명
    sources: string[];      // 출처 목록
  };
  safetyFlags: string[];    // 경고 사항 (있을 경우)
}

interface WorkoutDay {
  dayOfWeek: string;
  isRestDay: boolean;
  exercises: { name: string; sets: number; repsRange: string; note?: string }[];
}

interface DietDay {
  targetCalories: number;
  meals: { timing: string; foods: string[]; calories: number }[];
}
```

## 9. 기존 앱과의 통합 포인트

| 통합 대상 | 방향 |
|-----------|------|
| `user_profiles` / `user_goals` | 온보딩 시 기존 프로필 데이터 자동 pre-fill |
| `body_weights` | 체중 추이 → 정체기 감지, AI 컨텍스트 |
| `workout_sessions` | 완료율, 평균 볼륨 → AI 컨텍스트 |
| `meal_items` | 평균 섭취 칼로리 → AI 컨텍스트 |
| `user_programs` | AI 플랜 ≠ 기존 프로그램. 병행 가능, 충돌 알림 제공 |

### AI 플랜 vs 기존 프로그램 병행 규칙
- AI 플랜은 별도 플랜으로 저장 (기존 nSuns 등 프로그램 덮어쓰지 않음)
- 충돌 시 알림: "현재 nSuns 프로그램을 진행 중입니다. AI 운동 플랜을 함께 적용하면 볼륨이 중복될 수 있습니다. 계속하시겠어요?"
- 식단 플랜은 독립적으로 동작 (운동 프로그램과 충돌 없음)

## 10. 데이터베이스 추가 스키마 (신규 테이블)

```sql
-- AI 플랜 저장
CREATE TABLE ai_plans (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES auth.users NOT NULL,
  created_at TIMESTAMPTZ DEFAULT now(),
  week_start DATE NOT NULL,         -- 해당 주 월요일
  target_calories INT,
  target_protein INT,
  target_carbs INT,
  target_fat INT,
  plan_json JSONB NOT NULL,         -- AIPlan 전체 JSON
  is_active BOOLEAN DEFAULT true,
  generation_context JSONB          -- 어떤 데이터로 생성됐는지 메타
);

-- AI 기능 동의 현황
ALTER TABLE user_profiles
  ADD COLUMN ai_consent BOOLEAN DEFAULT false,
  ADD COLUMN ai_consent_at TIMESTAMPTZ;

-- RLS
ALTER TABLE ai_plans ENABLE ROW LEVEL SECURITY;
CREATE POLICY "users can manage own ai plans"
  ON ai_plans FOR ALL USING (auth.uid() = user_id);
```

## 11. 비용 추정 (참고)

| 시나리오 | 프롬프트 토큰 | 예상 비용/요청 |
|----------|--------------|---------------|

## 12. 리스크 및 트레이드오프

| 리스크 | 수준 | 대응 방안 |
|--------|------|-----------|
| LLM 응답 형식 불안정 (JSON 깨짐) | 중 | 응답 파싱 실패 시 재요청 + fallback 메시지 |
| 의료적 책임 | 고 | 면책 조항 + 위험 목표 거부 로직 |
| 부정확한 정보 입력 → 엉터리 계획 | 중 | "정확한 입력이 중요합니다" 강조 + 입력값 범위 검증 |
| 사용자가 계획을 무시 → AI 신뢰도 하락 | 중 | 완료율 기반 다음 주 계획 조정 + "계획 유지가 어려운 이유가 있었나요?" 피드백 수집 |
| 프라이버시 (LLM에 개인 데이터 전송) | 중 | 동의 기반 + 이름 등 PII 미포함 + 집계 데이터만 전송 |
| AI 플랜 ↔ nSuns 프로그램 충돌 | 저 | 명시적 충돌 알림 + 독립 저장 |
